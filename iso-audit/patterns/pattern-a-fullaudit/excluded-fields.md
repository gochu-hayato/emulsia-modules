# 機密フィールドの除外パターン

audit_logs や revisions に変更前後のドキュメントを保存する際、
機密情報をそのまま記録すると以下のリスクがある:

- 監査ログ自体が情報漏洩経路になる
- 暗号化前の状態でログに残る
- ログの閲覧権限を持つ人が機密情報にアクセス可能

## EBT 標準で除外していたフィールド

```typescript
EXCLUDED_FIELDS = [
  'password',
  'secret',
  'apiKey',
  'token',
  'refreshToken',
  'accessToken',
  'privateKey',
  'clientSecret',
];
```

## 案件で追加判断すべきフィールド

業務ドメインに応じて追加する例:

| 業務領域 | 追加すべきフィールド例 |
|---|---|
| 医療系 | 患者氏名、診断情報、保険番号 |
| 金融系 | 口座番号、カード番号、残高 |
| HR系 | マイナンバー、給与、評価 |
| 個人情報全般 | 住所、電話番号、メアド、生年月日 |

## 設計上の注意

- **「変更されたこと」自体は記録すべき**。値を伏せるだけ
- `changedFields`（フィールド名リスト）は残す
- 値の差分が必要なら「ハッシュ値の差分」で代替する設計もある

## ISO 27001 観点
情報資産分類で「機密」「秘」レベルのフィールドは
原則 audit_logs から除外する。
ログ自体のアクセス制御だけで守る設計は脆弱。
