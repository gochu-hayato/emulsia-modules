# ISO 9001 / 27001 対応履歴機能の要件

本ドキュメントは ISO 9001:2015 / 27001:2022 対応の履歴機能を実装する案件で満たすべき**最低限の要件**を定義する。実装方法は1つに固定しない。`patterns/` の参考実装またはカスタム設計から選ぶ。案件の事情に応じて柔軟に設計する。

---

## 関連 ISO 条項（参照）

- ISO 9001:2015 7.5.2（文書化情報の作成及び更新）
- ISO 9001:2015 7.5.3（文書化情報の管理）
- ISO 9001:2015 8.5.6（変更の管理）
- ISO 27001:2022 A.5.33（記録の保護）
- ISO 27001:2022 A.8.15（ログ記録）

各条項の正確な解釈は ISO 規格本体を案件側で参照のこと。

---

## 必須要件（すべて満たす必要あり）

### R1: 変更の完全記録
- すべての作成・更新・削除を記録する
- 誰が（userId）・いつ（timestamp）・何を（collection / documentId）・どう変えた（変更前後の値または差分）が分かる
- 変更されたフィールド名（changedFields）を識別できる

### R2: 真正性の確保
- 変更者は認証済みユーザーである
- 履歴データは追記のみ（append-only）または改ざん検知可能な構造
- Firestore セキュリティルールで write のみ許可、update/delete を禁止する

### R3: 変更理由の記録
- すべての更新・削除に `reason`（変更理由）フィールドを必須化する
- 監査時に「なぜこの変更を承認したか」を提示できる形で保持
- create は理由不要とする運用も可能（業務上の判断）

### R4: 重要文書の承認記録
- 案件ごとに「QMS 重要文書」コレクションを定義する（品質マニュアル・手順書・規定類等）
- 重要文書は承認なしの変更を許さない
- `approvedBy` / `approvedAt` / `approvalStatus` を記録

### R5: 機密フィールドの保護
- 履歴に機密情報（password / token / 個人情報等）を平文で残さない
- `patterns/pattern-a-fullaudit/excluded-fields.md` の方針に従って除外
- 「変更があった事実」と「フィールド名」は残す。値は伏せる or ハッシュ化

### R6: 検索可能性
- 監査時に「期間 × ユーザー × コレクション × 操作種別」で検索できる
- 監査人の要求に対して数秒以内に結果を提示できる

### R7: アクセス制御
- 履歴閲覧は管理者ロール限定（または案件ごとに定義した監査役割）
- 一般ユーザーへの開示範囲は案件で決定

### R8: 保持期間の明示
- 案件ごとに保持期間を定義し、ドキュメント化する
- 期間経過後の削除またはアーカイブ方針を案件で決定
- 推奨最低期間：3年（業界・業務により延長）

## 推奨要件（必須ではないが品質向上のため推奨）

### S1: 書き込み失敗時のリトライ
Cloud Functions の Firestore Trigger は書き込み前拒否ができない。失敗時に元データは残るが履歴だけ欠損する可能性がある。DLQ またはリトライキューの実装を推奨。

### S2: アトミック書き込み
複数の履歴コレクションに書く場合は Firestore バッチで一括書き込みを行い、部分成功・部分失敗を防ぐ。

### S3: 履歴閲覧 UI
監査効率化のため業務ユーザーが履歴を閲覧できる UI を提供する。タイムライン表示、フィルタ、CSV エクスポート等。

## Emulsia の哲学

ISO 要件を満たす実装方法は1つに固定しない。要件は満たしつつ、その会社の個性に合わせた残し方を案件ごとに設計する。「今いる人で最適化する」が Emulsia の根本姿勢。

SaaS 的に「全社統一の標準実装」を押し付ける発想は採らない。中小企業の個性こそが会社そのものであり、その個性を活かす履歴設計を案件ごとに提案する。
